## 完了までの順序

### 1. 現状確認と安全設計

1. `AGENTS.md` と正本資料、対象コード、現在のGit差分を読み、既存変更を失わないこと。
2. 個人情報を保存するコードを書く前に、緊急カードの脅威モデルを `docs/` に作ること。少なくとも、保護対象、想定攻撃者、端末紛失、端末ロック中、バックアップ、画面露出、ログ、PDF・印刷・共有、一時ファイル、削除、認証不能時を扱うこと。
3. 企画書の登録候補をそのまま全採用せず、30秒以内の緊急利用に必要な最小項目だけを決定すること。採用・不採用と理由を記録すること。
4. ローカル認証の有無と緊急時の可用性が衝突するため、既定動作、利用者の選択、ロック解除後の再認証、アプリが非アクティブになった際の秘匿を明記すること。
5. iCloud同期、アカウント、外部サーバー、解析、広告、常時位置情報は追加しないこと。

### 2. Slice 4「個人情報と出力」

脅威モデルで決めた境界に従い、次を一つずつ検証可能な縦スライスとして実装してください。

- 緊急カードの作成、閲覧、編集、削除。
- 緊急連絡先、集合・避難場所、常用薬・アレルギー等のうち、脅威モデルで採用した最小情報。
- 備蓄、お気に入り、記事データとは分離した保存モデルと移行計画。失敗を握りつぶさない明示保存。
- 端末内保護、必要なローカル認証、バックグラウンド移行時の画面秘匿、ロック時の再表示方針。
- 全データ削除と、削除後に再起動しても戻らないことの確認。
- VoiceOver、Dynamic Type、キーボード操作、ダークモード、高コントラストを壊さない入力・閲覧UI。

続けてPDF・印刷を実装してください。

- 緊急カードと備蓄・チェックリストから、利用者が含める項目を明示的に選択できるプレビュー。
- 個人情報を含むことを出力直前に説明し、利用者の明示操作なしに生成・共有・印刷しない。
- iPhone / iPadでは標準の共有・印刷導線、Macでは印刷向けプレビューと印刷導線を用意する。
- PDFへ入れない項目を確実に除外し、一時ファイル、ファイル名、ログ、エラー文へ個人情報を残さない。生成物の寿命と後片付けをテストする。
- オフラインで生成でき、失敗・キャンセル・空データを安全に扱う。

### 3. 残っているMVP導線

- 企画書の必須機能「公式情報リンク集」を、記事末尾の出典リンクだけで済ませず、オンライン接続が必要だと明示した独立導線として完成させる。
- リンク先は日本の公的機関を中心にし、`https`、名称、用途を機械可読に管理する。Web本文を転載せず、リンクを開く操作だけを通信境界にする。
- 30秒到達条件を満たすホーム導線を保つ。位置情報取得、リアルタイム通知、SNS、AI医療判断、クラウド共有は実装しない。

### 4. フルMVP品質ゲート

- Coreの単体・統合テストを追加し、正常系だけでなく、保存失敗、破損・欠損データ、認証キャンセル、削除、PDF除外、一時ファイル後片付けを確認する。
- SwiftDataはインメモリテストに加え、実ファイルの永続ストアで保存→コンテナ再生成→再取得と削除を確認する。
- UIテストまたは再現可能な手動スモーク手順で、ガイド、検索・フィルタ、お気に入り、備蓄、買い物、緊急カード、PDF・印刷を通す。
- 通信なしでのコールド起動と、ホームから最初の行動まで30秒以内をiPhone実機で計測する。
- VoiceOver、最大Dynamic Type、Reduce Motion、ダークモード、横向き、iPad、Macキーボード操作を確認し、結果を記録する。
- 日付変更をまたぐ期限表示、アプリ再起動、低電力状態、バックグラウンド復帰、空データ、保存容量不足に近い失敗を確認する。
- `draft` / `reviewed` / `approved` の検証を弱めず、配布候補で制作フィクスチャが監修済みに見えないことを確認する。

### 5. TestFlight準備

フルMVP品質ゲートを通してから着手してください。

- このMacへ必要なiOS Platformを導入し、iOS Simulator buildと主要フローを確認する。導入にユーザー操作が必要なら、そこで必要な操作だけを明示する。
- アプリアイコン、表示名、カテゴリ、対応端末・向き、バージョン、ビルド番号、Bundle IDを確認する。アイコンが未実装なら追加する。
- `project.yml` に開発チームを推測で書かない。利用可能なTeam、App ID、Automatic Signing、必要なDevelopment / Distribution証明書とProvisioning Profileを読み取りで確認し、不足分だけをユーザーへ依頼する。
- Release archiveを作成し、署名、entitlements、Bundle ID、埋め込みリソース、個人情報を含むデバッグログの不存在を検証する。
- Privacy ManifestとRequired Reason APIの使用有無を実コード・Archiveで監査し、必要な宣言だけを追加する。App Store ConnectのApp Privacy回答とプライバシーポリシーURLに必要な事実を文書化する。
- 暗号化・輸出コンプライアンス、TestFlightのベータ説明、テスト対象、フィードバックメール、内部テスター用スモーク手順を準備する。回答や連絡先は推測しない。
- まず内部TestFlightを対象にする。ArchiveのValidateまでは安全に進め、Uploadはユーザーの資格情報・明示的な実行権限が揃った場合だけ行う。

TestFlight候補では記事を増やさないでください。現在の `draft` フィクスチャは明確に未監修と表示し、将来のApp Store提出では `approved` 以外を製品コンテンツに混入させない配布ゲートを用意してください。

## 今回やらないこと

- 防災・医療・食品衛生の記事本文の本格拡充、監修、`approved` 化。
- iCloud同期、署名付きコンテンツ更新、完全オフライン地図、Apple Watch、課金、多人数共有。
- ユーザーが明示していないApp Store公開、外部TestFlight招待、証明書の失効・再発行。

記事の拡充・監修はTestFlight準備の次の独立フェーズです。App Store一般公開の最終条件には、全製品記事の出典、確認日、確認者、権利、実機アクセシビリティ確認が含まれます。

## 継続検証

各スライスで最低限、次を実行してください。

```sh
swift test --disable-sandbox
swift run --disable-sandbox content-lint
xcodegen generate                    # project.ymlを変更した場合
xcodebuild -project FourteenDayNote.xcodeproj -scheme FourteenDayNoteMac -destination 'platform=macOS' CODE_SIGNING_ALLOWED=NO build
xcodebuild -project FourteenDayNote.xcodeproj -scheme FourteenDayNote -destination '<利用可能なiOS Simulator>' CODE_SIGNING_ALLOWED=NO build
git diff --check
```

コマンドは実環境に合わせてDerivedData等を一時ディレクトリへ分離して構いません。失敗を「成功」と記録せず、環境要因とコード要因を分けてください。
